# Claude Project Notes

This file is for Claude or another coding agent picking up QuietCollective. Keep it factual and update it when project architecture, deploy flow, or permission rules change.

## Project Summary

QuietCollective is an AGPL-3.0-or-later private gallery, critique, and collaboration app for small artist communities. It runs on Cloudflare Workers with static assets, D1, private R2 media, and a queue-backed event/export workflow.

Core properties:

- Worker name: `quietcollective`
- Worker entrypoint: `worker/src/index.ts`
- Worker route registration: `worker/src/routes.ts`
- Frontend source: `web/src`
- Built frontend bundle: `public/app.js`
- Static assets and service worker: `public`
- D1 database binding: `DB`
- R2 media binding: `MEDIA`
- Queue binding: `JOBS`
- Cloudflare config: `wrangler.jsonc`
- OpenAPI spec: `public/api/openapi.yaml`
- Human-readable API docs: `public/developers.html` and `public/developers/index.html`

Do not commit secrets, `.dev.vars`, private domains, or private endpoint URLs.

## Local Setup

Install dependencies:

```bash
npm install
```

`wrangler` is a dev dependency, so `npm install` provides the project-pinned Wrangler binary.

For local Wrangler development:

```bash
cp .env.example .dev.vars
npm run d1:migrations:apply
npm run dev
```

The local dev server is `wrangler dev` through `npm run dev` or `npm run web:dev`.

Bootstrap the first admin by setting `JWT_SECRET` and `ADMIN_SETUP_TOKEN` in `.dev.vars` or Cloudflare secrets, then visit `/setup`. Setup is disabled after the first active admin exists.

## Required Commands

Use these before handing off changes:

```bash
npm run web:build
npm run worker:typecheck
npm run test:security
npm run test:permissions
npm run verify
```

`npm run verify` runs the build, worker typecheck, security regression tests, and permission tests.

Deployment:

```bash
npm run deploy
```

`npm run deploy` runs `npm run verify`, applies remote D1 migrations, then runs `wrangler deploy`.

If a deploy fails after `verify` but before `wrangler deploy`, rerun the failed step directly, then continue with the smallest safe command. Confirm live behavior with HTTP checks after deploying.

## Build Artifacts

Edit frontend source in `web/src`, not `public/app.js`. `public/app.js` is generated by `npm run web:bundle`.

`public/styles.css` and `public/sw.js` are source files, but `scripts/version-public-assets.mjs` rewrites version fingerprints in:

- `public/index.html`
- `public/sw.js`

Run `npm run web:build` before committing frontend/static changes so generated fingerprints and `public/app.js` are in sync.

`public/*.png` icons are committed intentionally. `public/assets/`, `.wrangler/`, `test-results/`, `.dev.vars`, and editor swap files are ignored or should be left untracked.

## Architecture Map

Worker:

- `worker/src/index.ts`: Hono app setup, auth/session helpers, capability helpers, event insertion/processing, queue consumer export.
- `worker/src/routes.ts`: HTTP routes and API handlers registered via `registerRoutes`.
- `worker/src/permissions.ts`: pure gallery/work capability logic. This is regression-tested directly.
- `worker/src/activity.ts`: activity and notification visibility joins.
- `worker/src/api-cache.ts`: ETag/cache token helpers for cacheable API reads.
- `worker/src/media.ts`: signed media token handling.
- `worker/src/web-push.ts`: direct Web Push VAPID implementation using `crypto.subtle`.
- `worker/src/archive.ts`: export zip building.
- `worker/src/types.ts`: Cloudflare env bindings and core row types.

Frontend:

- `web/src/main.ts`: lightweight client router and service worker registration.
- `web/src/app/*`: app state, API helper, shell, forms, notifications, comments, interactions.
- `web/src/pages/*`: route controllers that fetch API data and bind interactions.
- `web/src/views/*`: Preact TSX views rendered through islands.
- `web/src/components/*`: reusable UI, tiles, media, icons.
- `web/src/lib/markdown.ts`: markdown editor and rendering helpers.

Frontend rendering is mixed: route controllers produce strings and mount Preact island placeholders via `web/src/components/islands.tsx`. When adding a TSX view, add it to the island registry.

Database:

- D1 migrations live in `migrations/`.
- Add a new numbered migration for schema changes. Do not edit already-applied migrations unless explicitly resetting a local DB.
- Apply locally with `npm run d1:migrations:apply`.
- Apply remotely through `npm run deploy` or `npm run d1:migrations:apply:remote`.

## Permission Model

This is the highest-risk area. Read `worker/src/permissions.ts`, `scripts/work-permissions.test.mjs`, and the relevant security-regression checks before changing gallery/work access.

User-facing gallery ownership labels:

- Individual: stored as `ownership_type = "self"`.
- Collaborative: stored as `ownership_type = "collaborative"`.
- Everyone: stored as `ownership_type = "whole_server"` and/or `whole_server_upload = 1`.

User-facing visibility labels:

- Private: stored as `visibility = "private"`.
- Everyone: stored as `visibility = "server_public"`.

Important semantics:

- Everyone/server-public never means anonymous internet public. It means logged-in instance members.
- Individual galleries can be made visible to everyone, but only the owner can upload/edit/manage unless explicit code says otherwise.
- Individual gallery member rows may grant view/comment, but must not grant upload, edit, or manage capabilities.
- Collaborative gallery members can receive delegated upload/edit/manage capabilities.
- Everyone galleries allow any logged-in member to upload.
- Crosspost targets are allowed only when the target gallery is an Everyone gallery or is owned by the caller.
- Non-owned Individual and non-owned Collaborative galleries must not appear as crosspost targets.
- Work authors and explicit work collaborators can crosspost the work.
- Gallery upload permission alone does not grant work edit/version rights for someone else's work.
- Work collaborators can be credited for multiple roles. Do not reintroduce uniqueness by `(work_id, user_id)`; uniqueness is role-aware.
- Admin role does not grant content visibility or mutation. Admin should only enable explicit admin-page/admin-route operations.

Regression coverage:

- `scripts/work-permissions.test.mjs` compiles and tests `worker/src/permissions.ts`.
- `scripts/security-regression.test.mjs` asserts auth, visibility, notification, API docs, media, export, cache, and admin invariants by source inspection.

If intentional behavior changes, update tests in the same patch.

## Auth And Admin

Sessions use a signed `qc_session` cookie. Bearer tokens are also accepted by `requireUser`.

Admin bootstrap:

- `/api/setup/status`
- `/api/setup/admin`
- `/setup` frontend route

`ADMIN_SETUP_TOKEN` is required and the one-active-admin guard must remain. After bootstrap, registration is invite-only.

Admin pages manage instance/admin structures. Do not treat `role = "admin"` as content access bypass.

## Media Rules

R2 media is private. Do not make `quietcollective-media` public.

Media URLs must be minted only after the Worker has checked content permissions. When `R2_ACCOUNT_ID`, `R2_BUCKET_NAME`, `R2_ACCESS_KEY_ID`, and `R2_SECRET_ACCESS_KEY` are configured, `worker/src/media.ts` returns short-lived R2 S3 presigned GET URLs. Without those values it falls back to the app's signed Worker route.

- Signed media helper: `worker/src/media.ts`
- Signed media route: `/api/media/signed/:token`
- User media routes: `/api/media/users/...`
- Work/version media routes: `/api/media/works/...`

Originals and high-resolution files should not be cached by the service worker. `public/sw.js` has `isNeverCached()` checks for `/api/`, `/original`, and `high-resolution`.

New image uploads create preview and thumbnail variants. Existing media can be backfilled:

```bash
npm run media:backfill:webp -- --dry-run
npm run media:backfill:webp -- --remote
```

## Activity, Notifications, And Web Push

Domain events are inserted through `insertEvent()` in `worker/src/index.ts`. Event processing is queued when `env.JOBS` exists:

- `insertEvent()` writes to `domain_events`.
- It sends `{ kind: "process_event", eventId }` to `JOBS`.
- The queue consumer calls `processEvent()`.
- If queue send fails or no queue exists, it falls back to inline processing.

Notifications must be scoped to content the recipient can view. Mention notifications and activity reads should use visibility-aware helpers, not raw target IDs.

Web Push:

- Implemented directly in `worker/src/web-push.ts` with `crypto.subtle`; avoid adding heavy Web Push libraries that may not run in Cloudflare Workers.
- The push payload is a tickle, not the full notification payload. The service worker receives `push` and calls `/api/notifications/poll`.
- Subscriptions are managed by `/api/notifications/push-public-key` and `/api/notifications/push-subscriptions`.
- Failed/gone push subscriptions are disabled after repeated errors.

Browser fallback polling:

- First 30 minutes after activity: every 1 minute.
- Next 2 hours: every 5 minutes.
- Idle: every 30 minutes.
- When push subscription is active, Chrome/desktop should rely on push wakeups and use the idle fallback interval.

Notification click behavior:

- `public/sw.js` handles `notificationclick`.
- It marks the notification read, refreshes notification status, navigates/focuses a client, or opens a new window.
- App notification UI also updates the tab title with a red-dot prefix when unread notifications exist.

## API Cache And ETags

Cacheable API reads use `worker/src/api-cache.ts`.

Rules:

- Cacheable read routes should prepare the API cache before expensive DB reads.
- Mutating API requests bump the coarse API cache token.
- Do not cache auth/session-sensitive `state.me` data as a generic API JSON cache entry.
- Notification polling is ETagged and must not update member activity.

The service worker must never cache API responses.

## API Documentation

The human-readable docs are Redoc static pages:

- `/developers`
- `/developers/api`
- `/developers/`
- `/developers.html`

All render `/api/openapi.yaml`.

Important implementation detail: the Worker serves `/developers`, `/developers.html`, and `/developers/api` from the `/developers/` asset, not from `/developers.html`, to avoid Cloudflare static-asset pretty-URL redirect loops.

`wrangler.jsonc` must keep `"/developers*"` in `assets.run_worker_first`.

`scripts/check-public.mjs` requires:

- `public/api/openapi.yaml`
- `public/developers.html`
- `public/developers/index.html`

It also verifies `public/developers/index.html` matches `public/developers.html`. Keep those files in sync.

## UI Conventions

General UI:

- Main app routes are real app screens, not marketing pages.
- Keep controls compact and functional.
- Use existing icon helpers and component patterns.
- Do not add logout back to the top bar; it lives on the profile page.
- Sidebar "Install App" belongs near the bottom above admin links when present.
- Home page does not use a floating new-gallery plus button.

Visual details already requested by the product owner:

- Use "Individual", not "Self Owned", in user-facing text.
- Gallery and photo/work tiles need a very thin light border so they do not disappear on black backgrounds.
- Markdown editor top controls should fit on mobile.
- Markdown editor preview must be escapable.
- After inserting an image in markdown, leave the cursor at the end of the inserted image markdown.
- Markdown preview should show image thumbnails.

## Comments And Reactions

Comments can target galleries, works, versions, profiles, and comments. Reads and writes must be permission-checked against the target.

Notification behavior:

- Comments on gallery should use the header "Comments on Gallery".
- Comment push/action URLs should take the user to the relevant page and, where implemented, scroll/pulse the target comment.
- Likes/reactions on gallery, work, or comment should similarly target the visible UI element and use the red border pulse behavior.

Do not notify users about content they cannot view.

## Exports

Exports are queue/event backed and produce:

- JSON metadata
- Works CSV
- High-resolution WebP files
- Zip archive in R2

Exports expire and cleanup should delete undownloaded archive R2 keys. Security regression tests check archive keys, status, expiry, and cleanup.

## Source Control And Deploy Discipline

Before committing:

```bash
npm run verify
git status --short --branch
git diff --stat
```

Commit generated public asset changes when `npm run web:build` rewrites them. Do not commit editor swap files.

For deploys:

```bash

```

Then verify the live routes that changed. For API docs, check `/developers`, `/developers/api`, `/developers.html`, and `/api/openapi.yaml`.

## Common Pitfalls

- Do not make R2 public.
- Do not grant admin role implicit content access.
- Do not block public/server-visible Individual galleries; they can be visible to logged-in members while still owner-controlled for uploads.
- Do not let non-owners post directly to Individual galleries.
- Do not let non-owned Individual or Collaborative galleries appear as crosspost targets.
- Do not reintroduce a one-user-one-role limit for work collaborators.
- Do not send notifications for invisible tagged content.
- Do not cache API responses in the service worker.
- Do not remove `/developers*` from `run_worker_first`.
- Do not edit generated `public/app.js` directly.
